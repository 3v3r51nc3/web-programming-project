\section{Introduction}

EventHub is a full-stack web application project whose goal is to deliver a responsive Single Page Application (SPA) for managing events, participants, and registrations. The project must combine a modern frontend with a structured backend, and it must also document a technical comparison between two backend styles: a primary implementation in \textbf{Django} and a smaller comparative implementation in \textbf{Node.js and Express}. The report therefore addresses both software construction and software justification.

From a course perspective, the project is designed to connect the main web-programming layers in one coherent system. The frontend is responsible for interaction, responsive layout, navigation, and rendering. The backend is responsible for data modeling, validation, routing, and JSON responses. The two parts communicate through HTTP requests and REST-style API endpoints. The final application must show that these layers are not isolated topics from different lectures, but one integrated engineering workflow.

The architecture follows a clear full-stack separation. \section{Backend Design}

\subsection{Django as the primary backend}

Django is the main backend because the project is centered on structured relational data and on business rules that should be enforced consistently. Events, participants, and registrations form a classic business-domain model. Django is well suited for this because it provides models, migrations, validation, an administrative interface, and a predictable project structure. Instead of building every backend concern manually, the team can use Django's conventions to keep the code organized and maintainable.

\begin{center}
\begin{tabularx}{\textwidth}{@{}lY@{}}
\toprule
\textbf{Django element} & \textbf{Project role} \\
\midrule
Models & Define the database structure for events, participants, and registrations \\
Migrations & Apply schema changes safely to the database \\
Serializers & Convert model objects to JSON and validate incoming request payloads \\
ViewSets & Expose CRUD behavior with a compact REST-oriented structure \\
Router and URLs & Connect endpoints such as \texttt{/api/events/} to the correct logic \\
Admin & Provide a fast inspection and management interface during development and testing \\
\bottomrule
\end{tabularx}
\end{center}

The most important modeling decision is the explicit \texttt{Registration} model. This model does more than link two tables. It provides a place where business logic can live. For example, it can prevent duplicate registrations, record registration-specific metadata, and later support richer states such as approval or payment status.

\subsection{API organization and validation}

The API must expose the core resources of the application in a consistent way. The frontend should not depend on custom ad hoc endpoints when standard resource-based endpoints are sufficient. A compact REST structure therefore remains the clearest choice for the main project.

\begin{center}
\begin{tabularx}{\textwidth}{@{}l l Y@{}}
\toprule
\textbf{Method} & \textbf{Endpoint} & \textbf{Role} \\
\midrule
GET & \texttt{/api/events/} & Retrieve the list of events for the frontend \\
POST & \texttt{/api/events/} & Create a new event from validated form data \\
GET & \texttt{/api/participants/} & Retrieve participants and support administration tasks \\
POST & \texttt{/api/registrations/} & Create a new event registration after validation \\
\bottomrule
\end{tabularx}
\end{center}

Validation is a major backend responsibility. The frontend may offer helpful messages, but only the backend can be trusted as the final decision point. In EventHub this means the backend must reject invalid or duplicate registrations, enforce unique participant information where necessary, and return clear error messages when a rule is violated. This approach keeps the data trustworthy even if the frontend is bypassed or a malformed request is sent manually.

\subsection{Node.js and Express as the comparison backend}

Node.js and Express are kept in the project to make the backend comparison concrete. \textbf{Node.js} is the JavaScript runtime. \textbf{Express} is the lightweight framework that defines routes and HTTP behavior. In this project, the Node backend is intentionally smaller. Its role is to illustrate a more manual style of backend development rather than to replace the Django service.

\begin{center}
\begin{tabularx}{\textwidth}{@{}lYY@{}}
\toprule
\textbf{Criterion} & \textbf{Django} & \textbf{Node.js / Express} \\
\midrule
Project role & Main backend & Comparative backend \\
Structure & Strong defaults and conventions & Flexible and lightweight organization \\
CRUD productivity & Very strong for relational business apps & Good for smaller custom APIs, but more manual \\
Validation style & Model and serializer driven & Middleware and custom logic driven \\
Administration support & Built-in admin tools & Must be built manually or omitted \\
\bottomrule
\end{tabularx}
\end{center}

This comparison supports the report argument directly: Django is the better primary backend for EventHub because the application is model-heavy and validation-heavy, while Express remains useful as a teaching and comparison tool.

\section{Frontend Design}

\FrontendFigure

The frontend is a React SPA whose job is to provide a responsive interface for the main user workflows of the application. The frontend must not only display data, but also organize navigation, handle user input, show loading states, display validation feedback, and update the screen after API responses. That makes the frontend a real application layer rather than a simple HTML showcase.

\subsection{HTML, CSS, and React responsibilities}

The frontend is built on three complementary layers. HTML provides semantic structure. CSS provides visual layout and responsive behavior. React organizes the page into reusable components and keeps the visible interface synchronized with state changes. A weakness in any one of these layers reduces the quality of the final SPA.

The frontend design is organized around the main user tasks rather than around isolated widgets. Each page exists to support one clear operation in the project domain.

\subsection{Frontend-to-backend communication}

The frontend communicates with the backend through JSON-based HTTP requests. In practice, this means the React components must support at least three states whenever data is loaded from the API: loading, success, and error. This is a core part of the UI architecture because a professional SPA cannot assume that network requests always succeed immediately.

The same principle applies to forms. When a user submits data, the frontend should send a structured request, wait for the backend response, and then either update the screen or render a clear validation message. The frontend therefore acts as a disciplined API client, not as a source of truth about the data model.

\subsection{What each frontend technology does}

\textbf{HTML} answers the question: \emph{what is on the page?}

\textbf{JavaScript} answers the question: \emph{what happens when the user clicks, types, or requests data?}

\textbf{React} answers the question: \emph{how do we keep the interface synchronized with the current application state?}

\subsection{Step 9: understand \texttt{useState} as component memory}

State is what makes React feel alive. The easiest way to remember it is this: \texttt{useState} gives the component memory between renders. A normal local variable disappears after a render. State survives and tells React to render again when it changes.

\HooksMemoryFigure

\begin{lstlisting}[style=code, language=JavaScript, caption={One input and one boolean state}]
import { useState } from 'react'

function SearchPanel() {
  const [query, setQuery] = useState('')
  const [isOpen, setIsOpen] = useState(false)

  return (
    <section className="search-panel">
      <button onClick={() => setIsOpen(!isOpen)}>
        Toggle filters
      </button>

      {isOpen && (
        <input
          value={query}
          onChange={(event) => setQuery(event.target.value)}
          placeholder="Search events"
        />
      )}
    </section>
  )
}
\end{lstlisting}

\begin{center}
\begin{tabularx}{\textwidth}{@{}lYYY@{}}
\toprule
\textbf{Moment} & \textbf{query} & \textbf{isOpen} & \textbf{Visible result} \\
\midrule
First render & empty string & false & button only \\
User clicks button & empty string & true & input appears \\
User types ``par'' & ``par'' & true & input shows ``par'' \\
\bottomrule
\end{tabularx}
\end{center}

\LearningCallout{skywash}{
\textbf{Why this is memorable.}\\
\texttt{const [query, setQuery] = useState('')} creates one pair: the current state value and the function that updates it.\\
The component does not mutate the screen directly. It changes state.\\
React sees the new state and produces the new UI.\\
This is the core React sentence: \textbf{change state -> React re-renders -> UI updates}.
}

\ChecklistBox{
When you see \texttt{useState}, ask four questions in order. What value is stored. What event changes it. Which JSX reads it. What visual change appears after the update.
}

\MistakeBox{Do not write \texttt{query = 'par'} inside a component and expect React to update the screen. Direct assignment does not trigger React rendering. Use the setter function.}

\subsection{Step 10: understand \texttt{useEffect} and dependency arrays}

\texttt{useEffect} is for work that happens because the component exists or because some external value changed. Fetching data from the backend is the most common example in this project. The important memory rule is: rendering describes the screen, while \texttt{useEffect} handles side effects around the screen.

\begin{lstlisting}[style=code, language=JavaScript, caption={A safe loading effect with explicit states}]
import { useEffect, useState } from 'react'

function EventsPage() {
  const [events, setEvents] = useState([])
  const [loading, setLoading] = useState(true)
  const [error, setError] = useState('')

  useEffect(() => {
    let cancelled = false

    async function loadEvents() {
      try {
        setLoading(true)
        setError('')

        const response = await fetch('http://localhost:8000/api/events/')

        if (!response.ok) {
          throw new Error('Request failed')
        }

        const data = await response.json()

        if (!cancelled) {
          setEvents(data)
        }
      } catch (err) {
        if (!cancelled) {
          setError('Unable to load events')
        }
      } finally {
        if (!cancelled) {
          setLoading(false)
        }
      }
    }

    loadEvents()

    return () => {
      cancelled = true
    }
  }, [])
}
\end{lstlisting}

\begin{center}
\begin{tabularx}{\textwidth}{@{}lY@{}}
\toprule
\textbf{Dependency form} & \textbf{Meaning} \\
\midrule
\texttt{[]} & run once after the first render \\
\texttt{[query]} & run again when \texttt{query} changes \\
no dependency array & run after every render \\
\bottomrule
\end{tabularx}
\end{center}

\begin{lstlisting}[style=terminal, caption={Visible states the user can see}]
Loading events...
[ list of cards after success ]
[ Unable to load events ] after failure
\end{lstlisting}

\LearningCallout{mintwash}{
\textbf{How to memorize this effect.}\\
The first render happens with empty data.\\
Then the effect runs and asks the backend for JSON.\\
When the JSON arrives, the component updates state.\\
That state update triggers a new render with real data on screen.\\
So the sequence is: \textbf{first render -> effect -> fetch -> state update -> second render}.
}

\MistakeBox{The classic \texttt{useEffect} bug is an infinite loop. It appears when the effect changes a state value and that same value is also a dependency in a way that retriggers the effect forever.}

\subsection{Step 12: derive filtered UI instead of storing duplicate state}

One of the most important React habits is to store the smallest necessary state and derive the rest. In a search page, the real state is usually the original list plus the query. The filtered list is computed from those two values. It does not need its own permanent state variable.

\begin{lstlisting}[style=code, language=JavaScript, caption={A compact but realistic filtered page}]
import { useState } from 'react'

function EventsScreen({ events }) {
  const [query, setQuery] = useState('')

  const filteredEvents = events.filter((event) => {
    const haystack = `${event.title} ${event.location}`.toLowerCase()
    return haystack.includes(query.toLowerCase())
  })

  return (
    <main className="events-screen">
      <input
        value={query}
        onChange={(event) => setQuery(event.target.value)}
        placeholder="Search title or location"
      />

      <p className="helper-text">
        {filteredEvents.length} result(s)
      </p>

      <section className="event-grid">
        {filteredEvents.map((event) => (
          <EventCard
            key={event.id}
            title={event.title}
            location={event.location}
          />
        ))}
      </section>
    </main>
  )
}
\end{lstlisting}

\begin{lstlisting}[style=terminal, caption={Visible behavior}]
Query: "par"
Visible cards: events whose title or location contains "par"
Result count updates immediately after each keystroke
\end{lstlisting}

\LearningCallout{skywash}{
\textbf{Code reading focus.}\\
\texttt{query} is state because it changes over time.\\
\texttt{filteredEvents} is not state here; it is derived from \texttt{events} and \texttt{query}.\\
That makes the component easier to reason about because there is only one source of truth for the search text.
}

\MemoryBox{When React state starts to feel too big, ask: is this value truly stored, or can it be computed from existing state. That one question prevents many bugs.}

\subsection{Step 9: understand the serializer lifecycle}

Students often know that a serializer exists, but they do not know what job it actually performs. The serializer is the conversion and validation gate. Incoming request data enters as raw JSON-like data. The serializer checks it, transforms it into validated Python data, and then creates or updates model instances.

\begin{lstlisting}[style=code, language=Python, caption={A serializer with explicit business validation}]
class RegistrationSerializer(serializers.ModelSerializer):
    class Meta:
        model = Registration
        fields = '__all__'

    def validate(self, attrs):
        duplicate = Registration.objects.filter(
            event=attrs['event'],
            participant=attrs['participant'],
        ).exists()

        if duplicate:
            raise serializers.ValidationError(
                'This participant is already registered for this event.'
            )

        return attrs
\end{lstlisting}

\begin{center}
\begin{tabularx}{\textwidth}{@{}lY@{}}
\toprule
\textbf{Serializer stage} & \textbf{What it means in plain language} \\
\midrule
Raw input & request data arrives from the browser as JSON fields \\
Validation & the serializer checks required fields and business rules \\
Validated data & the backend now trusts the cleaned values \\
Save & Django creates or updates the model row \\
Response & the serializer turns the saved object back into JSON \\
\bottomrule
\end{tabularx}
\end{center}

\end{lstlisting}

from rest_framework.permissions import BasePermission, SAFE_METHODS

class IsEditorOrReadOnly(BasePermission):
    def has_permission(self, request, view):
        if request.method in SAFE_METHODS:
            return True

        return (
            request.user.is_authenticated
            and request.user.groups.filter(name='editor').exists()
        )
\end{lstlisting}

\subsection{Step 12: debug the backend without panic}

When a backend request fails, the right method is to debug in layers. Do not jump immediately into React. First verify that the server runs, then that the route exists, then that the request body is correct, then that validation passes, and only after that inspect the frontend integration.

\toprule

\midrule

\subsection{Memorable code-reading rules}

\begin{enumerate}
  \item In HTML, read from the biggest semantic block to the smallest element.
  \item In CSS, read selectors first, then layout properties, then spacing, then colors.
  \item In JavaScript, find the event source first: click, submit, input, timer, or fetch.
  \item In React, find the state variables before reading the JSX.
  \item In backend code, find the endpoint first, then the serializer, then the model.
  \item In debugging, trust visible evidence before guesses: status codes, console errors, and actual rendered output.
\end{enumerate}

\subsubsection*{4 analytical questions with model answers}

\begin{enumerate}
  \item \textbf{Question.} A component fetches data in \texttt{useEffect} but creates an infinite loop. Explain why and how to fix it.\\
  \textbf{Model answer.} The infinite loop happens when the effect updates a state value and that same state value appears in the dependency array. The state change causes a re-render, React detects a dependency change, and the effect runs again. If the effect should run only once on mount, use an empty dependency array. If it depends on a search term or id, then depend only on that true external input.\\
  \textbf{Explanation.} This is one of the most important analytical patterns in React effect management.

  \item \textbf{Question.} Compare REST and GraphQL for a large e-commerce application. Which would you choose and why?\\
  \textbf{Model answer.} REST is resource-based, easier to cache with standard HTTP tools, and usually simpler to scale operationally. GraphQL gives flexible queries and can reduce over-fetching in complex UIs. For a large e-commerce system with heavy traffic and strong caching needs, REST is often the safer default; GraphQL becomes attractive when the client requires highly variable nested data.\\
  \textbf{Explanation.} This answer emphasizes practical trade-offs rather than memorizing slogans.

  \item \textbf{Question.} Why does React use declarative rendering instead of imperative DOM manipulation?\\
  \textbf{Model answer.} Declarative rendering lets the developer describe the UI as a function of state. React then computes the necessary DOM updates. This reduces manual inconsistency, improves readability, and scales better because the UI stays synchronized with application state.\\
  \textbf{Explanation.} This is a core architectural reason for using React.

  \item \textbf{Question.} Why does separating API logic from UI components improve scalability?\\
  \textbf{Model answer.} It prevents fetch logic from being scattered across many components, centralizes error handling and token logic, and lets UI code focus on rendering. It also makes testing easier and allows several developers to work on the API layer and UI layer with less coupling.\\
  \textbf{Explanation.} This is a strong analytical answer because it connects architecture, maintainability, and team workflow.
\end{enumerate}

\subsection{Part B: 10 analytical questions and model answers}

\begin{enumerate}
  \item \textbf{Question.} Why does semantic HTML improve accessibility and SEO at the same time?\\
  \textbf{Model answer.} Semantic tags such as \texttt{header}, \texttt{nav}, \texttt{main}, and \texttt{section} describe the role of content instead of only its visual placement. Screen readers can navigate the structure more intelligently, and search engines can better understand which parts are titles, navigation, or content blocks.\\
  \textbf{Why this matters.} Semantic tags are not decoration; they are the foundation of good structure.

  \item \textbf{Question.} When should you choose Flexbox, and when should you choose Grid?\\
  \textbf{Model answer.} Choose Flexbox when the problem is one-dimensional, for example a navigation bar, a row of buttons, or centered content. Choose Grid when the layout needs rows and columns together, such as a dashboard, card layout, or content-plus-sidebar page.\\
  \textbf{Why this matters.} Confusing Flexbox and Grid often leads to harder CSS and weaker layouts.

  \item \textbf{Question.} Explain the difference between responsive and adaptive design.\\
  \textbf{Model answer.} Responsive design is fluid and adjusts continuously using flexible layouts, relative units, and media queries. Adaptive design relies more on fixed breakpoints and prepared layouts for selected screen sizes.\\
  \textbf{Why this matters.} Good CSS work requires design reasoning, not only syntax memorization.

  \item \textbf{Question.} Why can changing \texttt{width} or \texttt{margin} be more expensive than changing \texttt{color}?\\
  \textbf{Model answer.} Changes to width or margin can force layout recalculation, because the browser may need to recompute positions and sizes of elements. A color change usually affects painting only, so it does not rebuild layout.\\
  \textbf{Why this matters.} This explains why some CSS changes are visually cheap while others trigger expensive layout recalculation.

  \item \textbf{Question.} Why is JavaScript often placed at the end of the \texttt{body} or executed after DOM readiness?\\
  \textbf{Model answer.} If JavaScript runs before the DOM exists, selectors may fail because the elements are not yet available. Loading the script after the HTML or waiting for DOM readiness ensures that DOM operations work reliably.\\
  \textbf{Why this matters.} DOM timing bugs are common, so scripts must run only after the elements exist.

  \item \textbf{Question.} Why do we use \texttt{preventDefault()} in client-side validation?\\
  \textbf{Model answer.} By default, a form submission reloads the page. \texttt{preventDefault()} keeps the browser on the page so JavaScript can validate fields, show errors, and only submit when the data is acceptable.\\
  \textbf{Why this matters.} Validation messages must stay visible long enough for the user to understand the problem.

  \item \textbf{Question.} Why are keys required when rendering lists in React?\\
  \textbf{Model answer.} Keys give each rendered item a stable identity. This helps React compare the old list with the new list efficiently and avoid UI bugs when items are added, removed, or reordered.\\
  \textbf{Why this matters.} Keys improve both rendering correctness and performance, not just warning removal.

  \item \textbf{Question.} Explain why React prefers declarative rendering over manual DOM manipulation.\\
  \textbf{Model answer.} In declarative rendering, the developer describes the UI that should exist for a given state. React then computes the necessary DOM updates. This reduces inconsistency, improves readability, and scales better than manually editing DOM nodes step by step.\\
  \textbf{Why this matters.} This is one of the main architectural reasons React scales better than manual DOM manipulation.

  \item \textbf{Question.} Why can \texttt{useEffect} create an infinite loop?\\
  \textbf{Model answer.} An infinite loop appears when the effect updates a state variable and that same variable is also in the dependency array. The update triggers a re-render, the dependency changes, the effect runs again, and the cycle repeats.\\
  \textbf{Why this matters.} This is a classic analytical exam pattern, so the mechanism should be explained clearly and precisely.

  \item \textbf{Question.} Why is the \texttt{loading/error/success} pattern considered professional UI behavior?\\
  \textbf{Model answer.} Network requests are asynchronous and can succeed, fail, or still be in progress. A professional UI makes all three situations visible so the user always understands what is happening instead of seeing a blank or confusing screen.\\
  \textbf{Why this matters.} These three states are a serious frontend practice because they make asynchronous behavior understandable to the user.
\end{enumerate}

\textbf{Important.} The 10 practical tasks below are also fully corrected. The code block is the exact answer you can type in a practice file, the terminal block shows the visual or logical result, and the explanation tells you what to memorize for the exam.
}

\begin{enumerate}
  \item \textbf{Practical task 31.} Build the semantic skeleton of a correct main page.
\begin{lstlisting}[style=code]
<body>
  <header>
    <h1>EventHub</h1>
    <nav>
      <a href="events.html">Events</a>
      <a href="contact.html">Contact</a>
    </nav>
  </header>
  <main>
    <section>
      <h2>Upcoming events</h2>
      <p>Discover the next academic activities.</p>
    </section>
  </main>
  <footer>Author: Mehdi</footer>
</body>
\end{lstlisting}
\textbf{Expected visual result.}
\begin{lstlisting}[style=terminal]
+--------------------------------------+
| EventHub   Events   Contact          |
| Upcoming events                      |
| Discover the next academic activities|
| Author: Mehdi                        |
+--------------------------------------+
\end{lstlisting}
\textbf{Explanation.} This is the correct semantic base before any CSS is added.

  \item \textbf{Practical task 32.} Create an accessible login form with labels.
\begin{lstlisting}[style=code]
<form>
  <label for="email">Email</label>
  <input id="email" type="email" />

  <label for="password">Password</label>
  <input id="password" type="password" />

  <button type="submit">Login</button>
</form>
\end{lstlisting}
\textbf{Expected visual result.}
\begin{lstlisting}[style=terminal]
Email
[________________]
Password
[________________]
[ Login ]
\end{lstlisting}
\textbf{Explanation.} Correct form structure matters even before functionality is added.

  \item \textbf{Practical task 33.} Style a navigation bar with Flexbox.
\begin{lstlisting}[style=code]
nav ul {
  display: flex;
  justify-content: space-between;
  align-items: center;
  list-style: none;
  padding: 0;
}
\end{lstlisting}
\textbf{Expected visual result.}
\begin{lstlisting}[style=terminal]
Home                About                Contact
\end{lstlisting}
\textbf{Explanation.} Flexbox is ideal for horizontal alignment and navigation layout.

  \item \textbf{Practical task 34.} Create a two-column page layout with Grid.
\begin{lstlisting}[style=code]
.main-layout {
  display: grid;
  grid-template-columns: 3fr 1fr;
  gap: 20px;
}
\end{lstlisting}
\textbf{Expected visual result.}
\begin{lstlisting}[style=terminal]
+---------------------------+-----------+
| Main content              | Sidebar   |
+---------------------------+-----------+
\end{lstlisting}
\textbf{Explanation.} This is a classic content-plus-sidebar pattern built with Grid.

  \item \textbf{Practical task 35.} Show the visual difference between padding and margin.
\begin{lstlisting}[style=code]
.box {
  margin: 20px;
  padding: 16px;
  border: 2px solid #182620;
}
\end{lstlisting}
\textbf{Graphical result.}
\begin{lstlisting}[style=terminal]
outside gap = margin
+----------------------+
|  inside gap = padding|
|  content             |
+----------------------+
\end{lstlisting}
\textbf{Explanation.} This helps students reason visually about spacing instead of only memorizing definitions.

  \item \textbf{Practical task 36.} Add a responsive rule for mobile screens.
\begin{lstlisting}[style=code]
@media (max-width: 768px) {
  .main-layout {
    grid-template-columns: 1fr;
  }
}
\end{lstlisting}
\textbf{Expected result.}
\begin{lstlisting}[style=terminal]
Desktop:
[Main content][Sidebar]

Mobile:
[Main content]
[Sidebar]
\end{lstlisting}
\textbf{Explanation.} This is one of the most realistic CC1 CSS tasks because it combines Grid and media queries.

  \item \textbf{Practical task 37.} Change a title using DOM selection and a click event.
\begin{lstlisting}[style=code, language=JavaScript]
const title = document.querySelector("#hero-title");
const button = document.querySelector("#change-title");

button.addEventListener("click", () => {
  title.textContent = "Registration Open";
});
\end{lstlisting}
\textbf{Expected result.}
\begin{lstlisting}[style=terminal]
Before click: Welcome to EventHub
After click:  Registration Open
\end{lstlisting}
\textbf{Explanation.} Direct DOM interaction like this is the basis of more complex widgets.

  \item \textbf{Practical task 38.} Implement a FAQ accordion with show/hide logic.
\begin{lstlisting}[style=code, language=JavaScript]
const button = document.querySelector(".faq-question");
const answer = document.querySelector(".faq-answer");

button.addEventListener("click", () => {
  answer.classList.toggle("open");
});
\end{lstlisting}
\textbf{Expected visual result.}
\begin{lstlisting}[style=terminal]
Question: What is EventHub?
click -> answer opens
click again -> answer closes
\end{lstlisting}
\textbf{Explanation.} This is a standard FAQ accordion pattern.

  \item \textbf{Practical task 39.} Validate a contact form without reloading the page.
\begin{lstlisting}[style=code, language=JavaScript]
const form = document.querySelector("form");
const message = document.querySelector("#message");
const errorBox = document.querySelector("#message-error");
const successBox = document.querySelector("#form-success");

form.addEventListener("submit", (event) => {
  event.preventDefault();

  if (message.value.length < 10) {
    errorBox.textContent = "Message must contain at least 10 characters.";
    return;
  }

  successBox.textContent = "Sent!";
});
\end{lstlisting}
\textbf{Expected result.}
\begin{lstlisting}[style=terminal]
short message -> "Message must contain at least 10 characters."
valid message -> "Sent!"
\end{lstlisting}
\textbf{Explanation.} Good validation must show visible error messages and a clear success state.

  \item \textbf{Practical task 40.} Implement a light/dark theme toggle that persists after reload.
\begin{lstlisting}[style=code, language=JavaScript]
const themeButton = document.querySelector("#theme-toggle");

themeButton.addEventListener("click", () => {
  document.body.classList.toggle("dark");
  const nextTheme = document.body.classList.contains("dark")
    ? "dark"
    : "light";
  localStorage.setItem("theme", nextTheme);
});
\end{lstlisting}
\textbf{Expected design result.}
\begin{lstlisting}[style=terminal]
click -> body background becomes dark
reload page -> saved theme is restored
\end{lstlisting}
\textbf{Explanation.} This combines three important browser ideas at once: CSS classes, events, and localStorage persistence.
\end{enumerate}

\subsection{How to revise effectively}

\begin{enumerate}
  \item Start with Part A and answer the 10 QCM without looking at the solutions.
  \item Move to Part B and try to write each answer in 3 to 5 sentences.
  \item For Part C, rewrite the corrected code by hand so the patterns become automatic.
  \item For Part D, reproduce the practical solutions in a clean folder and test them in the browser.
  \item Revise JavaScript before React hooks, because React hooks depend on solid JavaScript reasoning.
  \item Memorize the three frontend layers as a chain: HTML gives structure, CSS gives layout and appearance, JavaScript gives behavior.
  \item Memorize the React chain as well: props bring data in, state stores changing values, effects connect the UI to asynchronous work.
  \item Memorize the API chain as one flow: client request, server response, JSON payload, and UI loading/error/success states.
\end{enumerate}
